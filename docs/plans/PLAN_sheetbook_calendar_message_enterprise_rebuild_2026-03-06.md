# PLAN: Sheetbook / Calendar / Message Enterprise Rebuild

작성일: 2026-03-06  
대상 서비스: `sheetbook`, `classcalendar`  
대상 사용자: 교사, 관리자, 학교 단위 운영자

## 1. 목표

이번 작업의 목표는 단순한 버그 수정이 아니다.

`sheetbook`, `calendar`, `message capture`를 교사 기준에서 하나의 자연스러운 업무 흐름으로 재정비하고,
기능 안정성, UX 명확성, 운영 검증 체계를 갖춘 엔터프라이즈급 서비스 기반으로 끌어올리는 것이다.

핵심 목표:

1. 어디서 시작하든 다음 화면이 예측 가능해야 한다.
2. 눌렀는데 안 되는 것처럼 느껴지는 버튼이 없어야 한다.
3. 기술 용어 없이도 교사가 흐름을 이해할 수 있어야 한다.
4. 결과가 저장되면 어디서 다시 보는지 즉시 알 수 있어야 한다.
5. 변경 이후에도 회귀를 빠르게 잡을 수 있는 검증 체계가 있어야 한다.

## 2. 완료 기준

아래 기준을 모두 만족해야 이번 리빌드가 완료된 것으로 본다.

### 2-1. 동선

1. `sheetbook -> calendar`
2. `message capture -> calendar event`
3. `sheetbook action -> consent/signature/notice`
4. `calendar -> 원본 서비스/첨부/수정`

위 4개 흐름이 모두 교사 관점에서 끊기지 않아야 한다.

### 2-2. UX

1. 주요 버튼은 모두 결과가 즉시 보이거나 다음 화면으로 바로 이어져야 한다.
2. "즉시 실행"과 "다음 화면 열기"는 문구와 배치에서 명확히 구분되어야 한다.
3. 화면은 도구 구조가 아니라 교사 작업 순서 구조로 읽혀야 한다.
4. `Ctrl+V`, `grid`, `A1:A1` 같은 숙련자 표현은 핵심 경로에서 제거하거나 약화한다.
5. 모바일/차단/실패 상황에서 무반응처럼 느껴지는 상태가 없어야 한다.

### 2-3. 운영/품질

1. `python manage.py check` 통과
2. 핵심 타깃 테스트 통과
3. 브라우저 스모크 통과
4. 콘솔 `ReferenceError` 0건
5. 주요 교사 시나리오를 문서화한 QA 체크리스트 보유

### 2-4. 엔터프라이즈 운영 가드

1. 키보드만으로도 주요 동선을 진행할 수 있고, 모달 닫기/포커스 복귀가 깨지지 않아야 한다.
2. 소유자/비소유자/관리자 권한 경계가 테스트로 확인되고, 연결 변경/생성 액션은 감사 가능한 흔적을 남겨야 한다.
3. 주요 생성/연결 실패는 로그와 메트릭으로 추적 가능해야 하고, 느린 화면/과도한 쿼리는 점검 대상에 포함돼야 한다.
4. 릴리즈 전에는 단계적 배포/롤백 기준과 서비스 매뉴얼, QA 체크리스트가 함께 준비돼야 한다.

## 3. 현재 진단

현재 기준에서 가장 큰 문제는 세 층으로 나뉜다.

### 3-1. 구조 문제

1. 진입점과 브리지가 뒤섞여 사용자가 기준 화면을 학습해야 했다.
2. 일부 연결은 명시적 설정이 아니라 휴리스틱에 의존한다.
3. `sheetbook`의 축약 캘린더와 `classcalendar` 본화면 간 기능 격차가 있다.

### 3-2. UX 문제

1. 버튼 라벨과 실제 동작이 달라 "버튼이 안 먹는 것처럼" 느껴지는 구간이 있었다.
2. 교사용 언어보다 도구/개발자 관점 언어가 앞에 노출된다.
3. 핵심 기능이 전면보다 숨겨진 위치에 배치되어 있다.

### 3-3. 품질 문제

1. 로컬 DB에 미적용 마이그레이션이 있어 브라우저 스모크가 붕괴돼 있다.
2. `sheetbook.tests`가 import 단계에서 깨져 있어 회귀 안전망이 불완전하다.
3. 수동 체감 문제를 자동으로 잡는 계약 테스트가 아직 부족하다.

### 3-4. 엔터프라이즈 격차

1. 키보드 이동, 포커스 복귀, 모달 닫기 같은 접근성 계약이 문서와 테스트로 충분히 고정돼 있지 않다.
2. 연결 구조 변경과 주요 생성 액션에 대한 권한 경계, 감사흔적, 운영 확인 포인트가 명확히 드러나지 않는다.
3. 성능, 관측성, 롤아웃 기준이 서비스 리빌드 계획 안에 직접 묶여 있지 않다.

## 4. 범위

### 포함

1. `sheetbook`
2. `classcalendar`
3. 두 서비스 간 연결 흐름, UI, 검증 체계

### 제외

1. 다른 앱 전반 리디자인
2. 공용 베이스 템플릿/전역 스타일 전면 개편
3. 신규 대형 제품 기능 추가

## 4-1. 리빌드 보강 축

이번 리빌드는 UI 수선만으로 끝내지 않는다. 아래 축을 모든 단계에서 함께 확인한다.

1. 접근성: 키보드 조작, 포커스 이동, 모달 닫기, 읽기 순서
2. 권한/감사흔적: 소유권 경계, 설정 변경 이력, 주요 액션 로그
3. 성능/관측성: 화면 응답성, 쿼리/에러 추적, 스모크 아티팩트 보존
4. 릴리즈/운영: 단계 배포, 롤백 기준, 서비스 매뉴얼/QA 문서 동기화

## 5. 단계별 실행 계획

## Phase 0. 완료 기준 고정

목표:

1. 교사 핵심 시나리오를 고정한다.
2. 각 시나리오의 합격 기준을 정한다.

핵심 시나리오:

1. 표 붙여넣기 -> 달력 반영
2. 메시지 붙여넣기 -> 일정 생성 -> 첨부 확인
3. 수첩에서 동의서/서명/안내문 열기
4. 수첩과 달력 간 왕복 이동

산출물:

1. 본 계획 문서
2. QA 체크리스트 갱신

## Phase 1. 환경/검증 복구

목표:

1. 현재 개발/검증 환경을 정상화한다.
2. 이후 UX 개편을 안전하게 진행할 수 있게 한다.

작업:

1. `core`, `classcalendar` 미적용 마이그레이션 적용
2. 브라우저 스모크 스크립트 재가동
3. `sheetbook.tests` import 하네스 정리
4. 스모크/테스트 실패 원인을 코드 문제와 환경 문제로 분리

완료 기준:

1. 스모크 스크립트가 DB 스키마 오류 없이 실행
2. `sheetbook.tests`가 import 단계에서 죽지 않음

## Phase 2. 연계 동선 정리

목표:

1. `sheetbook`, `calendar`, `message capture`가 한 흐름으로 이어지게 만든다.

작업:

1. 메인 진입점 단일 기준 유지
2. 생성 결과 노출 일원화
3. 원본 서비스 이동/첨부 열기/수정 경로 정리
4. 축약 캘린더와 본 캘린더의 기능 차이 축소

완료 기준:

1. 교사가 경로를 외우지 않아도 다음 화면을 예측 가능

## Phase 3. 정보구조 재설계

목표:

1. 현재 도구 중심 구조를 교사 작업 순서 구조로 바꾼다.

작업:

1. 주요 버튼 재배치
2. 핵심 기능 전면 노출
3. 고급/설정 기능 분리
4. 와이어프레임 흐름 재정렬

완료 기준:

1. 첫 화면만 봐도 "무엇을 먼저 해야 하는지" 이해 가능

## Phase 4. 교사용 언어/UI 정리

목표:

1. 기술 용어를 줄이고 교사 언어로 바꾼다.

작업:

1. 라벨/설명/토스트/오류 문구 재정비
2. 즉시 실행 vs 다음 화면 열기 구분 강화
3. 상태 피드백 통일

완료 기준:

1. 핵심 경로에서 숙련자 문법 없이도 이용 가능

## Phase 5. 명시적 연결 구조 도입

목표:

1. 최근 수첩/자동 추정 같은 휴리스틱 의존을 줄인다.

작업:

1. 어떤 수첩/탭이 연결되는지 사용자에게 보이게 함
2. 필요 시 연결 설정 UI 설계/구현

완료 기준:

1. 사용자가 "왜 여기로 연결되는지" 설명 가능

## Phase 6. 모바일/예외 UX 정리

목표:

1. 차단 상황과 예외 상황을 무반응처럼 느끼지 않게 만든다.

작업:

1. 모바일 read-only 정책 재검토
2. 차단 안내/폴백 동선 정리
3. 실패 시 다음 행동 안내 표준화

완료 기준:

1. 실패/차단 상태에서도 사용자가 다음 행동을 알 수 있음

## Phase 7. 디자인/최종 QA

목표:

1. 시각 완성도와 최종 운영 품질을 맞춘다.

작업:

1. 시각 계층/여백/상태 디자인 정리
2. 교사 여정 QA 문서 업데이트
3. 브라우저/모바일 최종 점검

완료 기준:

1. 기능, 동선, 문구, 시각 완성도가 함께 마감됨

## 6. 현재 우선순위

지금 당장 시작할 순서는 아래와 같다.

1. Phase 1 환경/검증 복구
2. Phase 2 연계 동선 정리 잔여분
3. Phase 3 정보구조 재설계
4. Phase 4 교사용 언어/UI 정리
5. Phase 5 명시적 연결 구조

## 7. 현재 알려진 리스크

1. 로컬 DB 미적용 마이그레이션이 브라우저 스모크를 막고 있다.
2. `sheetbook.tests` import 구조 붕괴가 이후 대형 개편의 회귀 방어를 약화시킨다.
3. 워크트리에 `products/*` 관련 사용자 변경이 있어 이번 범위와 분리 유지가 필요하다.
