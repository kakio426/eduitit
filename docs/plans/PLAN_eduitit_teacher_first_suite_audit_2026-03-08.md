# Eduitit Teacher-First Suite Audit

작성일: 2026-03-08
범위: `eduitit` 전체 서비스 진입 구조, 홈/카탈로그, 교사형 서비스의 첫 화면 UX, 서비스 통합 기준

## 1. 감사 기준

- 기본 사용자는 고급 디지털 도구 숙련자가 아니라, 빠르고 간단한 도구를 원하는 교사다.
- 첫 화면은 설명보다 현재 해야 할 일만 보여줘야 한다.
- 희소 기능, 관리 기능, 공유/협업/연동, 운영 메뉴는 기본적으로 접혀 있어야 한다.
- 동일 목표 CTA를 여러 군데에서 중복 노출하지 않는다.
- 전체 페이지 셸 중복, 브리지 화면, 중첩 스크롤, 과한 장식형 랜딩은 교사형 서비스의 최종 구조로 사용하지 않는다.
- 완료 기준은 "작동한다"가 아니라 "처음 쓰는 교사가 설명 없이 사용한다"이다.

## 1.5 기준선 잠금 상태

- `home_authenticated_v2.html` + `products/list.html` teacher-first 재설계는 현재 기준선 작업으로 취급한다.
- `noticegen`, `timetable`, `parentcomm`, `reservations` 첫 화면 표준화는 대표 서비스 셸 기준선으로 취급한다.
- `classroom_workspace + calendar + message`는 별도 정리된 canonical workspace 기준선으로 취급한다.
- 다음 전역 작업은 이 기준선 위에서만 진행한다.

## 2. 현재 구조 요약

- 활성 서비스 수: 42개
- 분류:
  - `classroom`: 12
  - `work`: 11
  - `game`: 9
  - `counsel`: 5
  - `edutech`: 3
  - `etc`: 2
- 루트 URL 네임스페이스 수가 많고, `config/urls.py` 기준으로 서비스가 평면적으로 노출된다.
- 실사용 교사형 서비스와 실험형/게임형/외부형 서비스가 홈과 카탈로그에서 충분히 분리되지 않는다.

## 3. 핵심 진단

### A. 전역 정보구조 문제

1. 홈이 교사 작업 공간이 아니라 서비스 전시장처럼 보인다.
- 근거: `core/templates/core/home_authenticated_v2.html`
- 검색, 클래스룸 워크스페이스, 즐겨찾기, 오늘 할 일, 목적별 섹션, 보조 섹션, 게임, SNS 위젯이 한 화면에 겹친다.
- `SNS 위젯`은 실시간 소통이라는 별도 제품처럼 강하게 노출되는데, 대부분 교사의 즉시 업무와 직접 연결되지 않는다.

2. 서비스 발견 경로가 중복된다.
- 홈 V2
- `products/list.html`
- `service_guide_list.html`
- `tool_guide.html`
- 일부 서비스별 자체 랜딩
- 사용자는 어디서 찾아야 하는지 다시 배워야 한다.

3. 카탈로그 언어와 제품 정체성이 teacher-first와 어긋난다.
- 근거: `products/templates/products/list.html`
- 영어 카피(`Digital Solutions`, `Play Now`, `Details`)와 쇼케이스형 카드 구조가 남아 있다.
- 교사 기준 "지금 뭐 하러 들어가야 하는지"보다 제품 진열이 먼저다.

4. 서비스 이름 체계가 목적보다 브랜드/별칭 중심인 경우가 많다.
- 예: `반짝반짝 우리반 알림판`, `최종최최종은 이제그만`, `몽글몽글 미술 수업`, `토닥토닥 선생님 운세`
- 내부 브랜딩은 유지할 수 있지만, 첫 화면에서는 작업명 중심 표기가 우선되어야 한다.

### B. 대표 교사형 서비스 문제

1. `classroom_workspace/classcalendar/message`
- 최근 정리로 많이 개선됐지만, 여전히 전체 제품 구조 안에서는 교사 작업의 canonical workspace로 충분히 승격되지 못했다.
- 홈에서 강하게 보이긴 하나, 다른 섹션/즐겨찾기/SNS와 같은 층위에서 경쟁한다.

2. `parentcomm`
- 근거: `parentcomm/templates/parentcomm/main.html`
- 한 화면에 `오늘 할 일`, `학부모 쪽지`, `상담 조율`, `알림장`, `연락처`가 모두 탭으로 들어가 있다.
- `상담 필요 등록 -> 방법 제안 -> 시간 추가`가 한 화면 안에 과도하게 노출된다.
- 교사 입장에서는 "오늘 처리"와 "설정/기록"이 분리되어야 한다.

3. `reservations`
- 근거: `reservations/templates/reservations/index.html`
- 화면 진입 직후 예약 모달 상태, 사용자 정보 기억 정책, 잠금 해제 정책, 관리자 링크, 날짜 네비게이션이 모두 전면에 드러난다.
- 실사용 흐름은 강력하지만, 첫 화면이 "예약하기"보다 "시스템을 이해하기"에 가깝다.

4. `timetable`
- 근거: `timetable/templates/timetable/main.html`
- 교사용 쉬운 언어를 쓰려는 방향은 맞지만, 첫 화면 설명/정책/점검 결과/예약 시스템 반영까지 한 화면에 너무 많이 쌓여 있다.
- `1단계`, `2단계`, `배치조건 입력 핵심`, `점검 결과`, `자동 배치`, `예약 반영`이 한 문맥에서 동시에 보인다.

5. `slidesmith`
- 근거: `slidesmith/templates/slidesmith/main.html`
- 실제 기능은 단순한데, `교사 발표 흐름`, `PDF 저장 안내`, 설명 카드, 상태 카드가 과하다.
- `입력 -> 미리보기 -> 발표`만 남겨도 되는 도구에 설명형 레이아웃이 많다.

6. `hwpxchat`
- 근거: `hwpxchat/templates/hwpxchat/main.html`
- 기능은 명확하지만, 첫 화면에서 `왜 이 도구를 쓰나요?`, `사용 방법 1~4`, HWP/HWPX 경고 등이 크게 노출된다.
- 좋은 도구지만 구조보다 설명으로 이해시키는 쪽으로 기울어 있다.

7. `happy_seed`
- 근거: `happy_seed/templates/happy_seed/dashboard.html`
- 첫 화면에 `교사용 설명서` CTA와 `빠른 시작` 박스가 전면에 보인다.
- 관리형 도구이지만, 첫 화면은 설명보다 `내 교실 열기 / 새 교실 만들기` 중심이 맞다.

8. `textbooks`
- 근거: `textbooks/templates/textbooks/main.html`
- 교재 생성과 자료실이 한 화면에 병치된 점은 강점이지만, `학생은 QR + 입장코드`, `PDF/텍스트 자료 모드` 등 운영 설명이 상단에서 크게 보인다.
- 자료 생성과 수업 진입을 더 강하게 분리할 여지가 있다.

9. `consent`
- 근거: `consent/templates/consent/dashboard.html`
- 비교적 안정적이다.
- 다만 정책 패널이 기본 노출일 경우 teacher-first 기준에서는 접힘 기본값을 고려할 수 있다.

10. `signatures`
- 근거: `signatures/templates/signatures/maker.html`
- 교사용 핵심 업무보다 `메이커` 제품 감성이 앞선다.
- 프리미엄 훅, 큰 미리보기, 꾸미기 요소가 업무형 UX와는 다소 다르다.

### C. 전사 공통 문제

1. 설명/가이드가 UI를 대신하는 화면이 여럿 있다.
- `tool_guide.html`
- `slidesmith/main.html`
- `hwpxchat/main.html`
- `happy_seed/dashboard.html`
- 일부 홈 위젯

2. 관리/운영 기능이 기본 화면에서 너무 자주 보인다.
- `SNS 위젯` 운영 메뉴
- `reservations` 관리자 대시보드 링크
- `classcalendar`의 공유/협업/연동 묶음
- `classroom_workspace/detail`의 탭 관리

3. 상위 탐색 구조가 과도하게 평면적이다.
- 교사형 핵심 도구와 실험형/게임형/외부형 도구가 같은 깊이로 나열된다.

4. 일부 서비스는 제품 브랜딩이 목적보다 앞선다.
- 브랜드는 보조로 남기고, 작업명을 전면으로 올려야 한다.

## 4. 우선순위별 수정 계획

### P0. 전역 IA 재설계

1. 교사 핵심 작업 축을 먼저 고정
- `수합·서명`
- `문서·작성`
- `학급 운영`
- `학부모 소통`
- `자료·발표`
- `교실 활동/게임`
- `상담·리프레시`

2. 홈을 `교사 오늘의 작업 공간`으로 축소
- 기본 화면:
  - 오늘 할 일
  - 최근 작업
  - 핵심 서비스 3~5개
- 하위:
  - 전체 서비스 보기
  - 가이드
  - 외부 서비스
  - 게임/실험실
- `SNS 위젯`은 홈 기본 배치에서 제거 후보

3. `products/list.html`를 teacher-first 카탈로그로 전면 교체
- 영어/쇼케이스형 UI 제거
- 작업명 중심 카드
- `지금 자주 쓰는 것 / 전체 / 게임 / 외부` 분리

### P1. 교사형 서비스 공통 셸 표준화

모든 교사형 서비스 첫 화면 공통 규칙:
- 제목
- 한 줄 목적
- 1~3개의 primary action
- 바로 작업 가능한 본문
- 공유/협업/설정/관리/설명은 접힘 또는 2차 위치

적용 우선 서비스:
- `classroom_workspace`
- `classcalendar`
- `parentcomm`
- `reservations`
- `timetable`
- `noticegen`
- `hwpxchat`
- `textbooks`

### P2. 설명형 화면 축소

대상:
- `slidesmith`
- `hwpxchat`
- `happy_seed`
- `tool_guide`
- `service_guide_list`

원칙:
- 큰 설명 카드 제거
- 사용 순서는 필요한 경우 1줄 또는 placeholder 수준으로 축소
- 더 자세한 가이드는 별도 도움말/매뉴얼로 이동

### P3. 브랜드명과 작업명 분리

- 카드/홈/검색/퀵액션에서는 작업명 우선
- 브랜드명은 서브 라벨로 유지
- 예:
  - `학급 캘린더` 유지 가능
  - `최종최최종은 이제그만` -> `문서 버전 관리`
  - `반짝반짝 우리반 알림판` -> `우리반 알림판`
  - `토닥토닥 선생님 운세` -> `오늘의 교사 리프레시`

### P4. 희소 기능 후퇴

- 공유
- 협업
- 연동
- 관리자 대시보드
- 운영 메뉴
- 개발성/실험성 옵션

이 기능들은 기본 화면에서 보이지 않아야 하며, `관리`, `더보기`, `설정`으로 모아야 한다.

## 5. 구현 체크리스트

- [ ] 홈에서 `지금 할 일` 외의 영역을 절반 이하로 줄인다.
- [ ] `products/list.html`를 teacher-first 카탈로그로 교체한다.
- [ ] `SNS 위젯`의 홈 기본 노출 여부를 재결정한다.
- [ ] `classroom_workspace/classcalendar`를 홈의 canonical workspace로 고정한다.
- [ ] `parentcomm`를 `오늘 처리 / 기록 / 설정` 구조로 재배치한다.
- [ ] `reservations` 첫 화면에서 예약 외의 상태/정책 노출을 줄인다.
- [ ] `timetable`를 `업로드 -> 점검 -> 배치 -> 반영` 단계형으로 분리한다.
- [ ] `slidesmith` 설명 카드 2개 이상 제거
- [ ] `hwpxchat` 설명 카드 2개 이상 제거
- [ ] `happy_seed` 설명서/빠른 시작 박스를 기본 화면에서 후퇴시킨다.
- [ ] 서비스 카드/검색/퀵액션에 작업명 우선 표기를 적용한다.

## 6. 검증 기준

### 정보구조
- 홈에서 핵심 작업 5개 이상이 한눈에 바로 구분된다.
- 게임/외부/실험형 서비스는 교사 핵심 업무보다 뒤로 밀린다.

### UX
- 첫 화면에서 긴 설명 박스가 없어도 주요 작업이 가능하다.
- 희소 기능이 주 기능보다 먼저 보이지 않는다.
- 중복 CTA가 없다.

### 스모크
- `홈 -> 핵심 서비스 진입` 5개 경로
- `수합·서명`
- `문서·작성`
- `학급 운영`
- `학부모 소통`
- `자료·발표`
- 각 경로에서 콘솔 `ReferenceError` 0

### 와이어프레임 기준
- 모든 교사형 서비스 첫 화면은 `제목 / 한 줄 목적 / primary action / 작업 본문` 구조를 따른다.
- 설정/공유/관리/운영은 첫 화면 하단 또는 접힘 구조여야 한다.
