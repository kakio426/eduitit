# AGENTS.md

## Rule Precedence

- 같은 주제의 규칙이 이 문서 안에서 서로 충돌하면, 더 아래에 있는 최신 규칙을 우선 적용한다.
- 새 규칙을 추가할 때는 기존 규칙을 무효화하는지 여부가 드러나도록, 충돌 가능성이 있는 더 오래된 규칙보다 아래에 둔다.
- 충돌을 발견했는데 바로 정리하지 못하면, 작업 중 해석은 `최신 규칙 우선`으로 고정하고 최종적으로 문서에도 반영한다.

## Branch Hygiene Policy

- 장기 유지 브랜치는 `main` 하나만 둔다.
- `feature/*`, `hotfix/*`, 서비스 전용 장기 브랜치는 새로 만들지 않는다.
- 코드 수정이 필요한 작업은 최신 `main`에서 짧은 수명 브랜치 `codex/<scope>`를 만들어 진행한다.
- 임시 브랜치는 한 작업 단위로만 사용하고, `main` 반영이 끝나면 로컬/원격 브랜치를 바로 삭제한다.
- 임시 `worktree`나 병합용 클론을 만들었으면 작업 종료 전에 반드시 제거한다.
- 현재 작업트리가 더럽거나 오래된 브랜치에 있으면 재사용하지 말고, 최신 `main` 기준의 새 작업 브랜치나 임시 작업 공간을 사용한다.
- 사용자가 `커밋만`이라고 말하면 임시 브랜치에 커밋만 남기고 `merge/push/branch 삭제`는 하지 않는다.
- 사용자가 `push만`, `이미 만든 커밋만 올려줘`, `이 커밋만 main에 반영해줘`처럼 이미 분리된 커밋의 반영만 요청하면, 새 개발 브랜치를 다시 만들지 말고 최신 `main` 기준의 깨끗한 통합용 `worktree`에서 `cherry-pick -> 검증 -> push -> cleanup`만 수행한다.
- 사용자가 임시 브랜치 보존을 명시하지 않았다면, `main` 반영 후 임시 브랜치를 남기지 않는다.

## Triggered Workflow (Korean Shortcut)

- 사용자가 아래처럼 요청하면 같은 규칙을 적용한다.
  - `main에 올려줘`
  - `main 반영까지 해줘`
  - `푸시까지 해줘`
  - `push해줘`
  - `작업 끝나면 브랜치 지워줘`
- 위 요청을 받으면 항상 아래 순서로 진행한다.
  1. `origin/main` 최신 상태를 먼저 확인한다.
  2. 코드 수정이 있으면 최신 `main`에서 `codex/<scope>` 임시 브랜치를 만든다.
  3. 요청 범위만 수정하고 테스트한다.
  4. `main`에 반영한다.
  5. 별도 지시가 없으면 `origin/main`까지 푸시한다.
  6. 임시 브랜치, 임시 `worktree`, 임시 클론을 바로 삭제한다.
- 사용자가 이미 분리된 커밋의 반영만 요청한 경우에는 아래 `push-only fast path`를 사용한다.
  1. `origin/main` 최신 상태를 먼저 확인한다.
  2. 현재 작업트리가 더럽거나 다른 작업 브랜치에 있으면, 그 작업트리는 건드리지 않고 최신 `main` 기준의 깨끗한 통합용 `worktree`를 만든다.
  3. 반영 대상 커밋만 `cherry-pick`하거나 필요한 최소한의 방식으로 가져온다.
  4. 요청 범위에 맞는 최소 검증만 다시 실행한다.
  5. `main`에 반영하고 `origin/main`까지 푸시한다.
  6. 통합용 임시 브랜치와 `worktree`를 바로 삭제한다.
- 사용자가 예전 표현인 `hotfix/main-ops로 진행해줘` 같은 요청을 하더라도, 이제는 `최신 main 기준 임시 브랜치에서 작업 후 main 반영/삭제`로 해석한다.

## Final Architecture Workflow

- 사용자가 아래처럼 말하면 기능 구현보다 `최종 제품 구조`를 우선한다.
  - `한번에 최종 구조로 가`
  - `엔터프라이즈급으로 만들어`
  - `직관적으로 만들어`
  - `설명 없이도 쓰게 해`
  - `중간 단계 없이 진행해`
- 위 요청이 있으면 아래 순서를 건너뛰지 않는다.
  1. 현재 UX 문제를 먼저 리뷰한다.
  2. 남길 화면 / 제거할 화면 / 통합할 화면을 정리한다.
  3. 텍스트 와이어프레임을 먼저 작성한다.
  4. 실제 구현 체크리스트를 먼저 만든다.
  5. 체크리스트 기준으로 구현한다.
  6. 구현 후 자기검수를 통과하기 전에는 완료라고 말하지 않는다.
- `작동은 한다`는 완료 기준이 아니다. 기본 완료 기준은 `설명 없이도 자연스럽다`이다.
- 중간 단계 구조, 과도기 UI, 임시 우회는 사용자가 명시적으로 허용하지 않으면 만들지 않는다.
- 설명이 많이 들어가야 이해되는 기능은 실패로 간주하고, 설명 추가보다 구조 수정부터 한다.

## Merge And Cleanup Rules

- `main` 반영 전에 `git status --short`와 브랜치 상태를 확인해 현재 작업 범위 밖의 변경이 섞이지 않게 한다.
- 이미 `main`에 들어간 변경을 다시 푸시하지 않는다. 먼저 `origin/main`과의 실제 차이를 확인한다.
- 작업 시작 전과 종료 전에 `git branch -vv`, `git worktree list`, `git rev-list --left-right --count origin/main...<branch>`로 각 브랜치가 `main`에 이미 들어갔는지와 어떤 `worktree`가 물려 있는지 확인한다.
- Python/Django 명령은 시스템 `python`/`python3`를 기본값으로 쓰지 않는다. 먼저 프로젝트 가상환경을 확인하고, 가능하면 그 인터프리터만 사용한다.
- 저장소 루트에 `.venv`가 있으면 `manage.py`, 테스트, Django import 확인, `pip` 확인은 기본적으로 `./.venv/bin/python` 기준으로 실행한다.
- 임시 `worktree`에는 `.venv`가 없을 수 있으므로, 이 경우 원본 저장소의 `.venv/bin/python` 경로를 명시적으로 사용한다.
- `ModuleNotFoundError`, `Django가 안 깔렸다`, `패키지가 없다` 같은 판단은 시스템 Python이 아니라 프로젝트 `.venv/bin/python`으로 다시 확인하기 전에는 단정하지 않는다.
- 오래된 로컬 변경이 `main`보다 뒤처진 상태면 재사용하지 말고 버릴지, 새로 적용할지 먼저 판단한다.
- 사용자의 미커밋 변경이 있는 더러운 작업트리는 보존한다. `main` 반영이나 `push`를 위해 그 작업트리를 강제로 `main`으로 돌리거나 정리하지 않는다.
- 더러운 작업트리에서 통합이 필요하면, 반영 작업은 별도 임시 `worktree`에서 수행하고 원래 작업트리에는 요청 범위 밖의 변경을 남겨 둔다.
- `main`에 이미 반영된 브랜치에서 계속 작업하지 않는다. `origin/main...<branch>` 기준으로 ahead가 `0`이면 새 작업은 그 브랜치에서 이어가지 말고, 필요 시 `main` 기준 새 임시 브랜치로 옮긴다.
- `main`에 이미 반영된 브랜치가 더러운 `worktree`와 함께 남아 있으면 바로 삭제하지 말고 먼저 dirty 파일을 `origin/main`과 비교한다. `origin/main`과 동일한 파일만 선택 정리하고, 실제 차이가 남는 파일은 `patch`나 별도 백업으로 분리한 뒤에만 `worktree/branch` 삭제를 진행한다.
- `UU` 같은 미해결 충돌 상태를 남긴 채 작업을 끝내지 않는다. 충돌을 유지해야 할 특별한 이유가 없다면 같은 세션에서 해결하거나, 최소한 백업 후 해당 `worktree`를 정리 대상으로 분류한다.
- 임시 패치 파일, 스모크 산출물, 병합용 메모 등은 보존 가치가 명확하지 않으면 `main`에 남기지 않는다.
- 문제 해결을 위해 만든 `scripts/fix_*`, `scripts/check_*`, `scripts/verify_*` 같은 일회성 스크립트는 기본적으로 임시 파일로 취급한다. 제품 코드나 재사용 가능한 운영 도구가 아니라면 작업 종료 전에 삭제한다.
- handoff/QA 문서, 스모크 JSON, 로그 캡처는 `제품 코드`와 분리해서 다룬다. 실제 인수인계나 운영 기록으로 남길 가치가 명확한 경우에만 수정하고, 그렇지 않으면 로컬 메모나 산출물로 보고 커밋하지 않는다.
- 스모크 실행 결과 파일은 기준 산출물로 보존하기로 명시되지 않았다면 커밋하지 않는다. 기본값은 `실행 후 삭제`다.
- 임시 검증용 스크립트나 메모를 만든 경우, 최종 커밋 전에는 `이 파일이 다음 작업에도 반복 사용될 운영 자산인가`를 먼저 판단한다. 답이 아니면 삭제한다.
- 원격 `fetch/push`가 필요한 작업은 네트워크나 권한 제약에 막히면 바로 승인 요청 또는 권한 상승 절차로 전환한다.
- 테스트가 길어질 가능성이 높으면 처음부터 충분한 `timeout`을 잡아 중간 타임아웃으로 흐름이 끊기지 않게 한다.
- `main` 반영이나 `push`가 끝난 뒤에는 통합용 `worktree`만 지우지 말고, `git fetch origin --prune`, `git branch -f main origin/main`, `git worktree prune`까지 확인해 로컬 `main`과 남은 `worktree` 메타데이터도 최신 상태로 맞춘다.
- 이미 `main`에 반영된 짧은 수명 브랜치는 기본적으로 오래 보관하지 않는다. 사용자가 보존을 명시하지 않았다면 `worktree` 정리 직후 로컬 브랜치도 함께 삭제한다.
- 작업 종료 시점에는 아래 상태를 목표로 한다.
  - 통합에 사용한 임시 작업공간의 `main` 또는 반영 대상 브랜치가 `origin/main`과 동기화됨
  - 사용자의 기존 작업트리가 더러웠다면, 그 작업트리는 원래 브랜치와 변경 상태를 유지함
  - 불필요한 임시 브랜치 없음
  - 불필요한 임시 `worktree`/클론 없음
  - 요청 범위 밖의 임시 스크립트/산출물 없음
  - 사용자가 정리를 원한 경우 최종 `git status --short`가 비어 있음

## Service Integration Guardrails

- 작업 시작 전에 반드시 `target_app`과 `do_not_touch_apps`를 먼저 선언한다.
- 작업 시작 첫 응답에는 이번 요청에 실제로 적용할 `AGENTS.md` 규칙, `target_app`, `do_not_touch_apps`, `이번 턴에서 안 건드릴 범위`를 함께 명시한다.
- `sheetbook`은 retired service로 취급한다. 새 계획, 추천, 파일럿, canonical workspace, teacher-first 기준선에 다시 포함하지 않는다.
- `sheetbook` 관련 작업은 기본적으로 `제거`, `격리`, `레거시 정리`만 허용한다. 새 기능 확장이나 재도입 후보로 다루지 않는다.
- 이 서비스의 기본 사용자상은 `AI나 디지털 도구를 능숙하게 다루지 않지만, 교실 업무를 빠르게 처리하고 싶은 교사`다. Notion/Google Sheets 숙련 사용자를 기준으로 정보 구조를 설계하지 않는다.
- 첫 화면은 `오늘 바로 하는 일`만 우선 보여 준다. 공유, 협업, 연동, 고급 설정, 관리자성 기능은 기본적으로 접거나 2차 위치로 내린다.
- 특별한 상황에서만 쓰는 기능이 주 작업보다 더 크거나 먼저 보이면 실패로 간주한다. 기본 화면에서 보조 기능이 주 기능의 주의를 빼앗지 않게 한다.
- 설명으로 기능을 이해시켜서는 안 된다. 컴맹에 가까운 교사도 버튼 위치와 화면 구조만으로 따라갈 수 있게 만든다.
- 서비스가 직관적이지 않고 구구절절한 설명에 기대면 실패로 간주한다.
- 설명으로 교사를 설득하려고 하지 않는다. 구조와 배치만으로 쓰이게 만든다.
- 설명이 필요하면 문구를 더 쓰지 말고 `UI`, `UX`, `와이어프레임`부터 다시 고친다.
- 사용법을 읽어야만 이해되는 화면은 실패다. 설명이 없어도 바로 눌러 쓸 수 있어야 한다.
- 좋은 서비스의 기준은 `잘 설명된 서비스`가 아니라 `설명이 필요 없는 서비스`다.
- 보조 설명은 교사가 화면 구조와 버튼만으로 해당 기능을 직관적으로 사용하기 어렵다고 판단되는 경우에만 넣는다.
- 화면 문구의 기본값은 `명확한 단어`, `짧은 상태`, `짧은 행동`이다.
- 제목, 버튼, 라벨, 상태값은 가능하면 한두 단어로 끝낸다.
- 본문 보조 문구가 꼭 필요해도 `한 줄`, `한 문장`을 넘기지 않는다.
- 한 카드/섹션 안에 설명 문단을 여러 개 두지 않는다. 설명이 길어지면 문구를 늘리지 말고 구조를 바꾼다.
- `~할 수 있습니다`, `~할 수 있어요`, `먼저 ~하세요`, `~하려면`, `~이 여기서 가능합니다` 같은 설명투를 기본 문체로 쓰지 않는다.
- 같은 뜻을 제목, 본문, 버튼에서 반복하지 않는다.
- 버튼은 설명형 문장 대신 행동만 남긴다. 예: `바로 시작할 수 있습니다` 대신 `시작`.
- 상태 문구는 해설 대신 결과만 쓴다. 예: `여기서 바로 확인할 수 있습니다` 대신 `확인 가능`.
- 섹션 제목 아래 긴 도입문을 기본으로 넣지 않는다. 제목과 첫 CTA만으로 이해되면 본문은 삭제한다.
- `1. ~ 2. ~ 3. ~` 식 사용 설명 블록은 기본 금지다. 정말 필요한 경우에도 각 줄은 초단문으로 쓴다.
- 운영 의도, 설계 이유, 사용법 해설은 사용자 화면이 아니라 문서/관리자 가이드로 뺀다.
- 요청 범위가 `global`이 아니면 공용 파일(`base.html`, 공용 CSS/JS, settings)은 수정하지 않는다.
- 시각 변경보다 동작 보존을 우선한다: 라우팅, 모달 닫기(배경/ESC), 포커스 복귀, 핵심 버튼 동작.
- 사용자가 스크린샷과 함께 `겹친다`, `튀어나온다`, `침범한다`, `가려진다` 같은 표현을 쓰면, 먼저 어떤 레이어(`상단바`, `콘텐츠 셸`, `카드`, `배경`)가 어느 영역을 침범하는지 문장으로 다시 특정한 뒤 수정한다. 문제를 비슷하지만 다른 레이아웃 이슈로 바꿔 해석하지 않는다.
- 고정/스티키 상단바가 있는 화면에서는 첫 콘텐츠 박스의 `배경`, `보더`, `그림자`가 상단바 안전영역 위로 올라오지 않게 한다. 상단 간격은 가능하면 내부 카드 패딩이 아니라 바깥 레이아웃 시작점에서 만든다.
- 공통 셸이 `--main-nav-height` 같은 실측 상단 높이 변수를 제공하는 경우, 서비스 진입 화면과 그 하위 주요 화면은 그 변수를 기준으로 첫 래퍼의 `padding-top` 또는 `top`을 잡는다. `pt-32`, `pt-28`, `margin-top: 88px`, `top: 7.25rem` 같은 고정 상단 보정값을 새 기본값으로 두지 않는다.
- 위 조건에 해당하는 템플릿에서는 첫 페이지 래퍼를 가능하면 `service-shell`, `service-shell--compact`, `service-shell--relaxed`, `service-shell--spacious`, `service-shell--hero` 중 하나로 통일하고, 인라인 `padding-top` 보정은 새 기본값으로 추가하지 않는다.
- 전체 화면 오버레이형 서비스라도 공통 상단바를 유지하는 구조라면 `fixed inset-0`로 셸 위를 덮지 않는다. 필요한 경우 시작 위치를 `top: var(--main-nav-height, ...)`로 내려서 상단바와 콘텐츠 레이어를 분리한다.
- 상단 간격 문제를 고쳤다면 해당 서비스의 진입 화면만 보지 말고, 같은 앱의 `목록 / 상세 / 생성 / 공유 / 규칙 / 플레이 / 완료`처럼 첫 래퍼가 따로 있는 화면도 함께 확인한다.
- UX 중심 작업에서는 구현 전에 `현재 문제 목록 -> 화면 통합안 -> 와이어프레임 -> 체크리스트`를 먼저 확정한다.
- UX 중심 작업에서 시각 버그를 고칠 때는 구현 전에 `무엇이 겹치는지`, `어느 박스가 잘못 올라왔는지`, `사용자 지적 요소와 실제 수정 대상이 일치하는지`를 한 줄씩 먼저 잠근다.
- UX 중심 작업에서는 구현 후 `기능 동작`과 별도로 `와이어프레임 / 버튼 중복 / 설명 과다 / 스크롤 구조 / 포커스 / 모바일`을 다시 점검한다.
- 같은 화면 안에서 같은 목적의 버튼을 여러 개 전면 배치하지 않는다. 하나는 주 버튼, 나머지는 보조 액션이나 숨김 메뉴로 내린다.
- 설정성 UI는 기본 화면의 주인공이 되면 안 된다. 매번 쓰지 않는 설정은 접거나 2차 위치로 내린다.
- 임베드나 호스트 화면에서는 상단바/푸터/헤더를 중복 렌더하지 않는다. 중첩 스크롤과 iframe 브리지는 최종 구조로 사용하지 않는다.
- `단일 서비스`, `단일 달력`, `단일 원본`이 요구된 작업에서는 canonical route/component/data source를 하나로 고정하고, 호스트 화면은 같은 본문을 직접 렌더하는 방향을 우선한다.
- 모바일은 `읽기 전용`과 `간단 편집 허용` 중 하나를 명시적으로 정한 뒤 구현한다. 막힌 기능은 비활성화하고 이유를 즉시 보여 준다.
- 구현 후에도 설명 문구가 많이 남아 있으면 구조가 덜 정리된 것으로 보고, 문구 삭제 가능 여부를 먼저 검토한다.
- 잔여 UX 문제가 남아 있으면 완료라고 단정하지 말고, 남은 항목과 영향도를 먼저 보고한다.
- 한글 깨짐 징후가 보이면 기능 수정보다 텍스트/인코딩 복구를 먼저 처리한다.
- `fetch` 기반 액션은 `response.ok` 검증과 실패 피드백(alert/toast)을 반드시 포함한다. 빈 `catch`는 금지한다.
- 폼 변경 시 required 모델 필드를 숨기면 안 된다. 숨길 경우 optional/default 처리 후 `form.errors`를 반드시 노출한다.
- JS 확인 UX(모달/2단계)에는 non-JS 제출 폴백을 유지해 "버튼 무반응" 상태를 만들지 않는다.
- 신규 서비스는 독립 앱 + URL namespace를 기본으로 한다.
- 신규 서비스 등록 시 `ensure_<app>`에서 `Product`와 함께 `ServiceManual(is_published=True)` 및 `ManualSection` 3개 이상을 동시에 생성한다.
- 머지 전 최소 검증:
  - 프로젝트 가상환경 Python으로 `manage.py check`
  - 예: 저장소 루트에서는 `./.venv/bin/python manage.py check`
  - 예: `worktree`에서는 원본 저장소 `.venv/bin/python manage.py check`
  - 변경 JS에 대한 `node --check`
  - 핵심 화면 런타임 스모크(주요 탭/버튼 1회) + 브라우저 콘솔 `ReferenceError` 0건
- 모델 변경, 신규 앱 추가, `migrations/*.py` 생성이 포함된 턴에서는 `manage.py makemigrations --check --dry-run`으로 누락 마이그레이션 0건을 확인한다.
- 신규 마이그레이션이 포함된 턴에서는 최종 보고에 `배포/서버에서 manage.py migrate --noinput 필요` 여부를 반드시 적는다. 자동 배포 훅에 `migrate`가 있어도 이 안내를 생략하지 않는다.
- 신규 앱 또는 신규 마이그레이션이 포함된 배포성 작업은 `manage.py showmigrations <app>` 또는 동등한 확인으로 대상 마이그레이션 상태를 확인하기 전에는 완료로 보지 않는다. 서버 접근이 불가능한 세션이면 남은 적용 확인 항목을 최종 보고에 명시한다.
